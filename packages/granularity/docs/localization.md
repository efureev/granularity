# Локализация

`@feugene/granularity` не навязывает приложению отдельный i18n-стек. Пакет использует локализацию только там, где у
компонентов есть встроенные интерфейсные строки, а источником правды для переводов остаётся само приложение.

Если нужен рекомендуемый i18n-слой для новой интеграции, базовый выбор — `@feugene/fint-i18n`: он хорошо ложится на
подход, в котором приложение полностью управляет локалью, словарями и переводческим контекстом, а `granularity`
использует этот общий слой без собственной изолированной i18n-инфраструктуры.

Это значит следующее:

- локаль, fallback-локаль и стратегия хранения выбранного языка определяются на стороне приложения;
- переводы продуктовых экранов и переводы дизайн-системы должны жить в одном общем i18n-слое приложения;
- если внешний перевод не подключён или нужного ключа нет, компонент использует свой fallback-текст и не ломает UI.

## Как это работает сейчас

Компоненты `granularity`, у которых есть встроенные подписи, кнопки или служебные тексты, обращаются к i18n-слою
приложения по ключу перевода.

Текущая схема поведения такая:

1. компонент запрашивает перевод по ключу вроде `gr.pagination.prev`;
2. если приложение возвращает нормальную локализованную строку, в UI используется она;
3. если i18n-слой не подключён или вернул исходный ключ, пакет использует встроенный fallback-текст.

Именно поэтому локализация в `granularity` — это не отдельный автономный модуль, а часть общей i18n-архитектуры хост-
приложения.

## Что важно учитывать

### 1. Локализация управляется приложением

`@feugene/granularity` не должен быть единственным местом, где создаётся i18n-инстанс. Правильнее, когда приложение:

- само определяет активную локаль;
- само решает, как хранить выбор языка;
- само объединяет свои словари и тексты дизайн-системы;
- само переопределяет package-level тексты там, где это нужно продукту.

Такой подход удобнее в реальном проекте: вся локализация остаётся централизованной, а пакет не начинает диктовать
структуру i18n-слоя.

### 2. Отсутствие перевода не ломает компонент

Если строка не найдена, компонент показывает встроенный fallback. Это полезно в двух сценариях:

- пакет подключают поэтапно, и словари ещё не полностью собраны;
- часть служебных текстов пока не вынесена в локализацию приложения.

Итог: интерфейс остаётся рабочим даже при частичной интеграции.

### 3. Переопределение текстов должно происходить в словарях приложения

Если нужно поменять подписи, кнопки, aria-label или служебные сообщения компонентов, правильное место для этого —
словарь приложения, а не форк пакета.

Так вы сохраняете единый контроль над терминологией и не усложняете обновление `@feugene/granularity`.

## Рекомендуемый подход

Наиболее здоровый сценарий такой:

1. если интеграция проектируется с нуля, рекомендуемый вариант i18n-слоя — `@feugene/fint-i18n`; если в приложении уже
   есть зрелый i18n-слой, `granularity` можно встраивать и в него;
2. переводы для `granularity` подключаются в этот же слой рядом с остальными словарями, например через package-level
   entrypoint `@feugene/granularity/i18n`;
3. при необходимости тексты дизайн-системы переопределяются на стороне приложения;
4. компоненты получают уже готовый перевод из общего i18n-контекста, а при его отсутствии используют fallback.

Это даёт предсказуемое поведение без дублирования i18n-инфраструктуры.

## Публичный API пакета

Пакет публикует package-level entrypoint `@feugene/granularity/i18n`, который можно использовать вместе с
`@feugene/fint-i18n` (начиная с `@feugene/fint-i18n@^0.3.0` — объявлена как optional peer dependency пакета;
более старые версии не тестировались и не гарантируются).

Он экспортирует:

- `GRANULARITY_I18N_BLOCK` со значением `gr`;
- `en` / `ru` / `es` — per-locale lazy loaders (рекомендуется для продакшена — бандлер вытряхивает неиспользуемые
  языки);
- `GRANULARITY_I18N_KEY` и связанные типы adapter API.

Загрузчики всех локалей сразу (демо/e2e/тулинг, не для продакшена) — отдельный subpath `@feugene/granularity/i18n/all`
(экспорт `all`).

Минимальный пример подключения:

```ts
import { createFintI18n } from '@feugene/fint-i18n/core'
import { installI18n } from '@feugene/fint-i18n/vue'
import { GRANULARITY_I18N_BLOCK, en, ru } from '@feugene/granularity/i18n'

const i18n = createFintI18n({
  locale: 'ru',
  fallbackLocale: 'en',
  loaders: [en, ru],
})

i18n.registerBlocks([GRANULARITY_I18N_BLOCK])
await i18n.loadUsedBlocks('ru')

// В точке входа приложения — без этого `granularity` не найдёт инстанс через
// provide/inject, и компоненты останутся на встроенных fallback-текстах.
installI18n(app, i18n)
```

Это не отменяет базовые правила интеграции:

- локализация принадлежит приложению;
- `granularity` ожидает внешний переводчик там, где он есть;
- при отсутствии перевода пакет использует встроенный fallback-текст.

## Локализация в companion-пакетах

Если вы разрабатываете **свой пакет-спутник** экосистемы (см.
[`companion-packages.md`](./companion-packages.md)) и у его компонентов есть встроенные строки, повторите
ту же схему, что и у ядра: `src/i18n` с per-locale загрузчиками, собственный **уникальный блок** (не `gr`),
экспорты `./i18n` и `./i18n/all`, `@feugene/fint-i18n` как optional peer, а компоненты читают перевод через
резолвер с fallback. Тогда потребитель подключит переводы вашего пакета **тем же** `createFintI18n`, что и
переводы ядра, — без отдельной i18n-инфраструктуры.

Пошагово это описано в разделе
[«Локализация компонентов через `@feugene/fint-i18n`»](./companion-packages.md#5-локализация-компонентов-через-feugenefint-i18n)
гайда по companion-пакетам; общий контракт публикации словарей — в документации `@feugene/fint-i18n`
(`docs/en/authoring-localization-packages.md`).

## Когда этот файл особенно полезен

Откройте этот раздел, если:

- хотите понять, почему компоненты работают даже без полной i18n-интеграции;
- планируете подключать `granularity` в приложение с уже существующим переводчиком;
- хотите переопределять встроенные тексты пакета без изменения его исходников.
