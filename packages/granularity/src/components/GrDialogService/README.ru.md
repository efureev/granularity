# GrDialogService (`useDialogService`)

## Что это

`GrDialogService` — это **императивный сервис диалогов** для дизайн-системы
Granularity. Он предоставляет
composable (`useDialogService`) и готовый синглтон (`dialogService`), которые
позволяют открывать диалоги подтверждения, оповещения и ввода
**прямо из секции `<script>` / `.ts`**, без размещения какого-либо компонента
диалога в шаблоне вызывающего компонента.

Внутри это тонкий оркестратор поверх существующих декларативных компонентов
`GrConfirmDialog` и `GrPromptDialog`. Он не переопределяет UI диалога, управление
фокусом или доступность — он переиспользует эти компоненты и лишь императивно
управляет их жизненным циклом.

## Зачем он нужен

Декларативные `GrConfirmDialog` / `GrPromptDialog` требуют от вызывающего кода
импортировать компонент, добавить его в шаблон, держать реактивное состояние
`modelValue` и подключать обработчики `@confirm` / `@cancel`. Для одноразовых
сценариев «спросил и забыл» (например, «подтвердить перед удалением») это
избыточно. Сервис убирает эту церемонию: единственный вызов возвращает `Promise`,
который разрешается решением пользователя.

Оба стиля рассчитаны на сосуществование:

- **Декларативный** (`<GrConfirmDialog v-model>`): когда диалог является частью
  состояния страницы, требует богатого кастомного контента через слоты или должен
  рендериться на сервере (SSR).
- **Императивный** (`useDialogService`): для быстрых одноразовых запросов,
  инициируемых из логики.

## Публичный API

- `useDialogService(defaults?)` — возвращает сервис, привязанный к опциональным
  общеприложенческим значениям по умолчанию. Также автоматически кэширует контекст
  приложения при первом вызове внутри `setup`.
- `dialogService` — заранее созданный синглтон-сервис без значений по умолчанию.
- Методы сервиса:
  - `confirm(message, options?)` — разрешается `true` при подтверждении, `false`
    в остальных случаях.
  - `alert(message, options?)` — диалог с одной кнопкой, разрешается при закрытии.
  - `prompt(message, options?)` — разрешается введённой строкой или `null` при
    отмене.
  - `open(kind, message, options?)` — низкоуровневый вызов, разрешающий
    дискриминируемый результат `{ action, value }`.
  - `closeAll()` — закрывает все активные диалоги и диалоги в очереди.
  - `setAppContext(ctx)` — явно регистрирует контекст Vue-приложения.
- Каждый метод возвращает `Promise`, дополненный методом `close()`, так что
  вызывающий код может закрыть диалог программно.

## Семантика разрешения (resolution)

Высокоуровневые методы никогда не реджектят; отмена — это штатный исход, а не
исключение. `confirm` разрешается булевым значением, `prompt` разрешается
`string | null`, а `alert` разрешается `void`. Потребителям, которым нужна точная
причина закрытия, следует использовать низкоуровневый `open()`, который
разрешается `{ action: 'confirm' | 'cancel' | 'close', value? }`.

## Асинхронный флоу подтверждения (`onConfirm`)

`confirm` и `prompt` принимают колбэк `onConfirm`, который получает управляющий
контекст и может быть асинхронным. Во время его выполнения:

- кнопка Confirm переходит в состояние загрузки/блокировки;
- закрытие по бэкдропу и Esc подавляется, чтобы диалог нельзя было закрыть «в
  полёте»;
- диалог остаётся открытым, если колбэк вернул `false`, выставил любую ошибку или
  бросил исключение.

Объект контекста предоставляет: текущее `value`, `AbortSignal`, который abort'ится
при закрытии диалога, `setError` / `setFieldError` / `clearErrors`, `setRawError`
(классификация ответа сервера, см. ниже), `setLoading` и `close`.

При успехе (ошибка не выставлена и не возвращён `false`) диалог закрывается, и
внешний промис разрешается результатом.

## Обработка ошибок сервера и поток данных

Ответы серверов бывают разной формы (Laravel, Symfony/RFC 7807, JSON:API, простые
Go-сервисы). Вместо ручного разбора `onConfirm` может передать сырой ответ
(`Response`, ошибку axios/fetch, `Error`, строку или JSON-объект) в
`ctx.setRawError`. Сервис прогоняет его через ту же цепочку парсеров, что
использует `GrResponseErrorBanner`, получая нормализованную ошибку:

- общее сообщение и детали рендерятся в `GrResponseErrorBanner` внутри тела
  диалога;
- `fieldErrors` мапятся на поля формы — для `prompt` ошибка поля `value`
  показывается под инпутом;
- диалог остаётся открытым, чтобы пользователь мог исправить ввод и повторить
  попытку.

Если `onConfirm` бросает исключение и ошибка не была выставлена вручную,
брошенное значение автоматически классифицируется через `setRawError`.

### Стратегия парсеров

Цепочка парсеров по умолчанию — **универсальное ядро** (`coreResponseErrorParsers`):
парсеры abort, network, HTTP-status и plain-message. Они работают на транспортном
уровне и безопасны для любого бэкенда; они не делают формат-специфичных
допущений, поэтому никогда не «ложно срабатывают» на почти-Laravel ответах.

Серверо-специфичные парсеры (валидация Laravel, problem details RFC 7807,
JSON:API) подключаются осознанно (opt-in). Опция `errorParsers` принимает либо
массив полной замены, либо builder-функцию, которая получает именованные пресеты
(`core`, `all`, `laravel`, `problemDetails`, `jsonApi`, `fileValidation`) для
удобной композиции, плюс место для кастомных парсеров. Обычно это настраивается
один раз в `useDialogService(defaults)` и может быть переопределено для каждого
вызова.

## Контекст приложения (i18n / тема / provider)

Поскольку хост монтируется императивно (вне обычного дерева компонентов), он
должен наследовать контекст приложения, чтобы работали i18n, темизация и granular
provider. Приоритет разрешения:

1. опция `appContext` на уровне вызова;
2. явно зарегистрированный контекст через `setAppContext`;
3. авто-кэшированный контекст, захваченный при первом запуске `useDialogService()`
   внутри `setup` компонента.

Авто-кэш позволяет сервису работать «из коробки» в большинстве приложений; явная
регистрация доступна для вызывающего кода вне `setup` и для конфигураций с
несколькими приложениями / микрофронтендами.

## Очередь

Диалоги сериализуются через FIFO-очередь. В любой момент рендерится только голова
очереди; последующий вызов показывается после закрытия предыдущего диалога. Это
исключает конкуренцию за focus-trap и наложение бэкдропов, а также делает
поведение предсказуемым, когда несколько диалогов инициируются почти одновременно.

## Жизненный цикл

Единственный компонент-хост лениво монтируется в `document.body` при первом
использовании и переиспользуется на протяжении всего времени жизни приложения.
Каждый запрос диалога создаёт `AbortController`, который abort'ится при закрытии,
и запрос удаляется из очереди после разрешения, позволяя открыться следующему
диалогу из очереди. Сервис безопасен для SSR: методы являются no-op, когда
`window` / `document` недоступны.

## Доступность и UX

Все гарантии доступности обеспечиваются переиспользуемым стеком `GrConfirmDialog`
/ `GrPromptDialog` / `GrDialog` / `GrModal`: единственный заголовок диалога,
focus-trap и восстановление фокуса. Закрытие по Esc и бэкдропу следует опциям
`closeOnEsc` / `closeOnBackdrop` и разрешается через ветку cancel/close. Во время
асинхронного подтверждения основное действие сообщает о прогрессе через своё
состояние загрузки.
