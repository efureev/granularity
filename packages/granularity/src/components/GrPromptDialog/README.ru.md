# GrPromptDialog

## Что это

`GrPromptDialog` — декларативный примитив диалога ввода одного значения для
дизайн-системы Granularity, построенный поверх `GrDialog`. Он показывает
заголовок, опциональное описание, поле ввода с подписью (через `GrFormField` +
`GrInput`) и действия Confirm / Cancel. Значение ввода доступно через
`v-model:value`, а видимость диалога — через `v-model`.

Как и `GrConfirmDialog`, это тонкий слой: механика оверлея, управление фокусом и
доступность приходят из нижележащего стека `GrDialog` / `GrModal`. Компонент
добавляет поле ввода, поведение required/валидации, опциональный блок серверной
ошибки и пропсы асинхронного подтверждения, которые использует императивный
`useDialogService`.

## Зачем он нужен

Запросить у пользователя один короткий текст («назовите проект», «укажите
причину») — постоянно повторяющаяся потребность. `GrPromptDialog` стандартизирует
этот флоу — раскладку, подпись поля, правило «обязательное» и отображение
ошибок — чтобы prompt-диалоги вели себя одинаково и доступно без переписывания на
каждом экране.

Это декларативный аналог `useDialogService.prompt`. Оба стиля сосуществуют:

- **Декларативный** (`<GrPromptDialog v-model v-model:value>`): когда prompt
  является частью состояния страницы, требует кастомного контента тела через слоты
  или должен рендериться на сервере (SSR).
- **Императивный** (`useDialogService.prompt`): для быстрых одноразовых запросов
  ввода, инициируемых из логики. Сервис монтирует и управляет этим компонентом под
  капотом.

## Публичный API

- `v-model` (`modelValue`) — управляет видимостью.
- `v-model:value` (`value`) — строка ввода (обязательный проп).
- Контентные пропсы: `title`, `description`, `label`, `placeholder`.
- Пропсы действий: `confirmText`, `cancelText`, `confirmVariant`, `confirmTone`,
  `buttonSize`.
- Пропсы валидации: `required` (по умолчанию `true`), `requiredErrorText`,
  `fieldError` (внешняя/серверная ошибка поля, имеет приоритет над встроенной
  проверкой required).
- Проксируемые в `GrDialog` пропсы: `size`, `closeOnBackdrop`, `closeOnEsc`,
  `showHeader`, `showCloseButton`, `headerConfig`, `footerConfig`, `bodyConfig`,
  `closeLabel`.
- Async/сервисные пропсы: `error` (`ResponseErrorInfo`, рендерится через
  `GrResponseErrorBanner`), `confirmLoading`, `confirmDisabled`, `closeOnConfirm`.
- События: `confirm` (несёт значение), `cancel`, `update:modelValue`,
  `update:value`.
- Слоты: дефолтный (кастомное тело над полем), `error` (кастомный блок общей
  ошибки), `footer` (кастомные действия).

## Поведение валидации

Когда `required` равно `true` (по умолчанию), пустое значение подсвечивается
ошибкой только после того, как поле было «затронуто» — то есть после blur или
попытки Confirm — чтобы диалог не выглядел ошибочным сразу при открытии. Клик
Confirm при пустом обязательном значении не закрывает диалог: он помечает поле как
затронутое и ставит фокус на ввод.

Внешняя `fieldError` (обычно из серверной валидации) имеет приоритет над встроенной
проверкой required и показывается под полем ввода, при этом поле помечается как
невалидное.

## Обработка ошибок сервера и поток данных

Клик Confirm эмитит `confirm` с текущим значением. Если `closeOnConfirm` равно
`true` (по умолчанию), диалог закрывается; при `false` закрытие делегируется
вызывающему коду — именно так `useDialogService` держит диалог открытым, пока
выполняется async `onConfirm`.

Для серверных ошибок сервис классифицирует сырой ответ в `ResponseErrorInfo`.
Общее сообщение показывается через опциональный проп `error`
(`GrResponseErrorBanner`), а ошибка уровня поля для поля `value` мапится на
`fieldError` и рендерится под полем ввода — позволяя пользователю исправить
значение и повторить попытку, не теряя контекст.

## Значения по умолчанию и i18n

Тексты по умолчанию — «Confirm» / «Cancel», лейбл «Value», сообщение об
обязательности «Enter a value.» и лейбл кнопки закрытия «Close». Это обычные
пропсы, локализуемые передачей переведённых строк; императивный сервис подключает
i18n-aware дефолты автоматически.

## Доступность и UX

Гарантии доступности приходят из `GrDialog` / `GrModal` (единственный заголовок,
focus-trap, восстановление фокуса) и из `GrFormField` / `GrInput` (связь
label/error, состояние invalid). При открытии шум валидации подавляется до первого
взаимодействия. Во время асинхронного подтверждения кнопка Confirm показывает
состояние загрузки, а закрытие подавляется, чтобы диалог нельзя было закрыть «в
полёте».
