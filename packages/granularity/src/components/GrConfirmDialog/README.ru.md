# GrConfirmDialog

## Что это

`GrConfirmDialog` — декларативный примитив диалога подтверждения для дизайн-системы
Granularity, построенный поверх `GrDialog`. Он показывает заголовок, сообщение (или
богатое кастомное тело) и два действия — Confirm и Cancel — сообщая о решении
пользователя через события и управляя видимостью через `v-model`.

Это намеренно тонкий, сфокусированный слой: вся механика оверлея, управление
фокусом, переходы и доступность приходят из нижележащего стека `GrDialog` /
`GrModal`. `GrConfirmDialog` лишь добавляет раскладку действий confirm/cancel,
опциональный блок серверной ошибки и пропсы асинхронного подтверждения, которые
использует императивный `useDialogService`.

## Зачем он нужен

Подтверждение — один из самых частых паттернов диалога («вы уверены, что хотите
удалить?»). Переписывать раскладку «заголовок/тело/две кнопки» для каждого такого
случая утомительно и чревато ошибками. `GrConfirmDialog` стандартизирует раскладку,
тона кнопок, тексты по умолчанию и доступность, чтобы подтверждения выглядели и
вели себя одинаково по всему продукту.

Это декларативный аналог императивного `useDialogService`. Оба стиля рассчитаны на
сосуществование:

- **Декларативный** (`<GrConfirmDialog v-model>`): когда диалог является частью
  состояния страницы, требует богатого кастомного контента через слоты или должен
  рендериться на сервере (SSR).
- **Императивный** (`useDialogService.confirm`): для быстрых одноразовых сценариев
  «спросил и забыл», инициируемых из логики. Сервис монтирует и управляет именно
  этим компонентом под капотом.

## Публичный API

- `v-model` (`modelValue`) — управляет видимостью.
- Контентные пропсы: `title`, `description` (используется, когда дефолтный слот
  пуст).
- Пропсы действий: `confirmText`, `cancelText`, `confirmVariant`, `confirmTone`,
  `buttonSize`.
- Проксируемые в `GrDialog` пропсы: `size`, `closeOnBackdrop`, `closeOnEsc`,
  `showHeader`, `showCloseButton`, `headerConfig`, `footerConfig`, `bodyConfig`,
  `closeLabel`.
- Async/сервисные пропсы: `error` (`ResponseErrorInfo`, рендерится через
  `GrResponseErrorBanner` в теле), `confirmLoading`, `confirmDisabled`,
  `closeOnConfirm`.
- События: `confirm`, `cancel` и `update:modelValue`.
- Слоты: дефолтный (кастомное тело, заменяет `description`), `error` (кастомный
  блок ошибки, по умолчанию — `GrResponseErrorBanner`) и `footer` (кастомные
  действия).

## Поведение и поток данных

- Клик Confirm эмитит `confirm`. Если `closeOnConfirm` равно `true` (по
  умолчанию), диалог также закрывается через `update:modelValue=false`. При
  `false` закрытие делегируется вызывающему коду — именно так `useDialogService`
  держит диалог открытым, пока выполняется async `onConfirm`.
- Клик Cancel (или закрытие по бэкдропу / Esc / кнопке закрытия) эмитит `cancel`
  и закрывает диалог.
- Проп `confirmLoading` переводит кнопку Confirm в состояние загрузки для
  асинхронного подтверждения «в полёте»; `confirmDisabled` принудительно
  дизейблит её.

## Обработка ошибок сервера

Опциональный проп `error` принимает нормализованный `ResponseErrorInfo`. Когда он
выставлен, компонент рендерит незакрываемый `GrResponseErrorBanner` внутри тела.
Это в первую очередь используется `useDialogService`, чтобы показать
распознанные серверные ошибки (сетевые сбои, валидация и т. д.), не закрывая
диалог и позволяя пользователю повторить попытку. Слот `#error` даёт полностью
кастомное отображение ошибки.

## Значения по умолчанию и i18n

Тексты действий по умолчанию — «Confirm» / «Cancel», лейбл кнопки закрытия —
«Close». Это обычные пропсы, поэтому приложения локализуют их, передавая
переведённые строки (императивный сервис подключает i18n-aware дефолты
автоматически).

## Доступность и UX

Гарантии доступности наследуются от `GrDialog` / `GrModal`: единственный заголовок
диалога, focus-trap и восстановление фокуса на инициатор при закрытии. Закрытие по
Esc и бэкдропу следует опциям `closeOnEsc` / `closeOnBackdrop` и разрешается через
ветку cancel. Во время асинхронного подтверждения основное действие сообщает о
прогрессе через своё состояние загрузки, а диалог подавляет закрытие, чтобы его
нельзя было закрыть «в полёте».
