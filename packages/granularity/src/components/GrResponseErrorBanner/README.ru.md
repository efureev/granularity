# GrResponseErrorBanner

## Что это

`GrResponseErrorBanner` — транспортно-независимый компонент отображения ошибок для
дизайн-системы Granularity. Он рендерит нормализованную структуру
`ResponseErrorInfo` — общее сообщение, опциональные строки по полям/деталям и бейдж
HTTP-статуса — как тонированный `GrAlert`. Он **не** делает запросов и ничего не
знает про axios/fetch/xhr; он лишь отображает уже распарсенную структуру ошибки.

Пакет поставляет три слоя вокруг одного движка:

- компонент `GrResponseErrorBanner` (отображение);
- composable `useResponseError` (stateful-хелпер classify/retry/dismiss);
- подключаемый конвейер парсеров, превращающий сырые ответы любой формы в
  `ResponseErrorInfo`.

Также есть две «мнениевые» обёртки: `GrUploadErrorBanner` (пресеты для загрузки
файлов с включённым retry) и `GrFormErrorBanner` (пресеты для валидации форм с
подписями полей и тоном warning для валидации).

## Зачем он нужен

Серверные и сетевые ошибки приходят в разных формах (валидация Laravel, problem
details RFC 7807 / Symfony, JSON:API, простые Go-сервисы, голые `Error`, сетевые
`TypeError`). Отрисовка каждой из них «на месте» приводит к несогласованному,
непереведённому и трудно тестируемому UI ошибок. Этот компонент централизует и
*классификацию* (через парсеры), и *отображение* (через единый тонированный
alert), чтобы состояния ошибок выглядели одинаково и легко локализовались и
тестировались по всему продукту.

## Поток данных

1. Сырая ошибка/ответ нормализуется (`normalizeError`) в единообразную форму
   (статус, распарсенное тело, заголовки, флаги abort/network) независимо от
   транспорта.
2. Цепочка парсеров (`ResponseErrorParser[]`) анализирует этот контекст; результаты
   сливаются по порядку, и парсер может прервать цепочку (`stop`). Итог —
   `ResponseErrorInfo` с семантическим `kind`, сообщением, опциональными `details`
   и структурированными `fieldErrors`.
3. Баннер рендерит эту информацию: производные от `kind` тон и заголовок,
   сообщение, дедуплицированный список деталей/полей, опциональный бейдж статуса и
   кнопки действий.

## Категории ошибок

`ResponseErrorKind` классифицирует ошибки по семантике, что задаёт тон по
умолчанию: `network` (ответа нет — транспортный сбой), `aborted` (отменено
клиентом), `validation` (400/422 или структурированные ошибки полей — «исправь и
повтори»), `client` (прочие 4xx), `server` (5xx — retry разумен) и `unknown`
(fallback). По умолчанию `aborted` авто-скрывается, поэтому отмены проходят тихо.

## Публичный API

- Пропсы компонента включают: `error` (структура ошибки, `null` скрывает баннер),
  `texts` (частичный override текстов), `tone` / `toneByKind`, `showDetails`,
  `showFieldLabels`, `fieldLabels`, `dedupeDetails`, `canRetry`, `canDismiss`,
  `autoHideKinds`, `showStatus`, `testIdPrefix`.
- События: `retry` (несёт текущую ошибку) и `dismiss`.
- `useResponseError(options)` возвращает: `currentError`, `isVisible`, `lastRaw`,
  `setRaw` (классификация + сохранение, async), `setError` (сохранить готовую
  info), `classify` (классификация без сохранения), `dismiss` / `reset` и `retry`
  (выполнить handler и сбросить при успехе).
- Парсеры и хелперы: `coreResponseErrorParsers` (универсальное ядро),
  `defaultResponseErrorParsers` (полная цепочка), `responseErrorParserPresets`,
  `extendDefaultParsers`, `createResponseErrorClassifier`, `normalizeError`,
  отдельные парсеры (`laravelValidationParser`, `problemDetailsParser`,
  `jsonApiErrorParser`, `fileValidationParser`, `httpStatusParser`,
  `networkErrorParser`, `abortErrorParser`, `plainMessageParser`).

## Стратегия парсеров

Универсальное ядро (`coreResponseErrorParsers` — abort, network, HTTP-status,
plain-message) работает на транспортном уровне и безопасно для любого бэкенда; оно
не делает формат-специфичных допущений, поэтому никогда не «ложно срабатывает» на
почти-Laravel ответах. Серверо-специфичные парсеры (валидация Laravel, problem
details RFC 7807, JSON:API, валидация файлов) подключаются осознанно (opt-in) и
компонуются через `responseErrorParserPresets` или `extendDefaultParsers`.
Кастомные парсеры — это простые функции `(ctx) => result | null`, их можно
вставить в любое место цепочки.

## Тексты и i18n

Заголовки и сообщения по каждому `kind`, а также лейблы retry/dismiss, имеют
встроенные fallback-тексты. Они разрешаются через слой переводов granularity
(`gr.responseError.*`) и могут быть дополнительно переопределены через проп
`texts`. Если парсер оставил английское дефолт-fallback сообщение, баннер
подменяет его i18n-переводом соответствующего kind.

## Доступность и UX

Баннер построен поверх `GrAlert`, наследуя его семантику и стилизацию тона. Детали
рендерятся списком с опциональными подписями полей; дублирующиеся строки между
основным сообщением и деталями удаляются. Retry и dismiss — опциональные кнопки,
которые эмитят события, а не выполняют побочные действия, оставляя управление
вызывающему коду. Обёртки `GrUploadErrorBanner` и `GrFormErrorBanner` просто
предзадают эти опции под свои сценарии.
