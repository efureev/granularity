# E2E: a11y (axe) + визуальная регрессия (Playwright)

Сквозной слой качества для дизайн-системы. Витрина рендерит живые демо всех
компонентов (включая ручные WAI-ARIA паттерны: `GrSelect`, `GrAutocomplete`,
`GrSlider`, `GrTree`, `GrTabs`, `GrDropdown`), поэтому именно она — площадка для
автоматических a11y/visual проверок.

## Запуск

```bash
# всё сразу (собирает dist библиотеки + генерит контракты, затем гоняет тесты)
yarn test:e2e

# по слоям
yarn test:a11y             # только axe
yarn test:visual           # только визуальная регрессия
yarn test:visual:update    # пересоздать эталонные скриншоты
yarn test:e2e:ui           # интерактивный UI-режим Playwright
```

Первый запуск скачивает браузер: `npx playwright install chromium`.

Тесты поднимают dev-сервер vite (base `/`), который отдаёт SPA на глубоких
маршрутах и использует собранный `dist` библиотеки через alias. Список компонентов
берётся из `src/content/generated/componentApi.generated.json` — новый компонент
попадает под оба слоя автоматически.

## a11y (`a11y.spec.ts`)

Прогоняет `axe-core` по области `[data-example-preview]` (сам отрендеренный
компонент, без хрома витрины) по каждому компоненту.

**Модель гейта — baseline на регрессии.** Для компонента берём serious/critical
нарушения, вычитаем зафиксированный долг (`a11y-baseline.ts`) и падаем на остатке.
Значит ловим: регрессии в «чистых» компонентах, новые компоненты без a11y и рост
долга. Долг выжигается удалением строк из baseline по мере починки.

- `color-contrast` отключён на уровне axe: он завязан на дизайн-токен `--muted-fg`
  по всей системе (лейблы/подписи/hint'ы) — это отдельный трек («Токены как данные»),
  а не структурный баг компонента; изменения цвета ловит визуальный слой.
- Компоненты `GrSlider`, `GrAutocomplete`, `GrTabs`, `GrTree`, `GrDropdown`,
  `GrModal` и др. проходят гейт **начисто** (их нет в baseline).

## Визуальная регрессия (`visual.spec.ts`)

Снимок области «Live examples» для репрезентативной выборки компонентов в **light и
dark** темах (тема форсится до загрузки приложения). Эталоны — в
`e2e/__screenshots__/`, коммитятся в git. Первый прогон создаёт их, дальше —
сравнение с допуском на антиалиасинг. Цель — ловить непреднамеренные сдвиги
токенов/стилей, а не покрыть каждый пиксель.

> Скриншоты платформо-зависимы (шрифты/рендер). Эталоны сгенерированы под
> `chromium-darwin`; для стабильного CI генерируйте их в том же окружении (или в
> Docker-образе Playwright) через `yarn test:visual:update`.
